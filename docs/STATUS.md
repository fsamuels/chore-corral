**Completion notes, behind a confirm dialog (2026-07-19).** Marking a chore **Done** no longer applies instantly — from a user ask that it's too easy to complete a chore by accident, and that completion should be able to carry an optional note about how it went. **Reminder snooze: 10 minutes or 1 hour (2026-07-19).** Follow-on to the same-day reminders feature below, from a direct user ask. A fired reminder can now be snoozed — it **mutates the same `task_reminders` row** (`remind_at` = now + N, `sent_at` back to null; no new row, no migration) so it re-fires through the existing every-minute pipeline and shows as upcoming again in the chore's Reminders card (see DECISIONS.md for row-mutation-over-insert and the rest of this entry's reasoning). Two surfaces, because iOS web push renders no notification action buttons at all: **(1) on the notification** (Android/desktop Chrome) — `public/sw.js` now offers "Snooze 10 min" / "Snooze 1 hr" action buttons (only when the payload carries the new `reminderId` field, added to the Edge Function's payload contract) and, on tap, POSTs to `/api/reminders/snooze` without opening a window, then replaces the notification (same tag) with a no-actions confirmation — or, on failure, a "Couldn't snooze — open the chore to try again." notification that keeps the chore URL tappable; **(2) in-app** (the iOS path) — already-sent reminder rows in `TaskReminders.vue` gain the same two buttons (compact 32px outline pills, sharing the row's existing in-flight disable tracking). The SW path required the project's **first Nitro server route**, `server/api/reminders/snooze.post.ts`: a service worker has no Supabase session (can't read auth cookies from JS), but its same-origin fetch carries the cookies, so the route rebuilds the user's own session via `serverSupabaseClient`/`serverSupabaseUser` (`#supabase/server`) and updates **under RLS** — no service-role key, no authorization logic of its own; 400 on bad input, 401 unauthenticated, RLS-invisible rows map to 404. Shared pure logic in new `app/utils/reminder-snooze.ts` (`SNOOZE_OPTIONS`, strict `isValidSnoozeMinutes` whitelist guard, `snoozeTargetIso`) is used by the route, the new `snoozeReminder` service (`app/services/reminders.ts`, wired through `useTaskReminders` — which follows the codebase's no-`.single()`-after-`.update()` convention), and the UI. Built with a Sonnet subagent, coordinator-reviewed. **Tests:** new `tests/reminder-snooze.test.ts` plus `snoozeReminder` cases in `tests/reminders.test.ts` (397 Vitest cases total); no FakeSupabaseClient changes needed. `eslint`, `prettier --check`, `nuxt typecheck`, and `pnpm build` (now bundling the server route) all clean. **Deploy note:** the Edge Function payload changed, so the not-yet-done first deploy of `send-reminders` (Next Steps item 40) picks this up automatically — no extra step; snooze verification was folded into that same item. **Task completion attribution generalized to multiple people via a new `task_completers` join table (2026-07-15).** The prior shape — `tasks.completed_by` (uuid) XOR `tasks.completed_by_name` (text), added 2026-07-12 — could only ever credit one person per task. **Schema** (`supabase/migrations/20260715120000_task_completers.sql`): a new `task_completers` table (`id`, `task_id` FK → `tasks` `ON DELETE CASCADE`, `user_id` FK → `auth.users` nullable, `completer_name` text nullable), with a per-**row** `CHECK ((user_id IS NULL) <> (completer_name IS NULL))` replacing the old task-level XOR — a task's completer set can now freely mix app members and free-text names, since only each row is constrained, not the set's composition. Two partial unique indexes (`task_completers_task_user_uniq`, `task_completers_task_name_uniq`) prevent crediting the same member or the same free-text name twice on one task. RLS mirrors `task_tags`: no `farm_id` column, scoped through the parent task, `auth.uid()` wrapped in a scalar subselect. The migration backfills both old columns into the new table, then drops `completed_by`, `completed_by_name`, and the `tasks_completed_by_xor_name` constraint outright. **Service** (`app/services/completers.ts`, new): `setTaskCompleters` (replace-a-task's-whole-set, patterned on the tags service) and `listCompletersForTasks`, plus an `assertValidCompleters` app-layer XOR guard mirroring the dropped `assertCompletedByXorName`. **Behavior:** marking a task done now auto-credits the acting member only when the task has zero completers already (previously it unconditionally overwrote the single `completed_by`); leaving done still deletes the whole completer set, in step with `completed_at`. **UI:** the task detail page's "Completed by" control (`app/pages/tasks/[id]/index.vue`) is now a multi-select chips picker (any number of members plus free-text names) instead of a single member-or-name field; the Progress page (`app/pages/progress.vue`) renders every completer on a completed task's row instead of just one. **Seed data**: `supabase/seeds/clarksons_farm_sample_data.sql` now inserts into `task_completers` instead of the old columns, including one task (`325d87d2-89a0-47ec-9095-69010dbff2ac`, the burst-pipe fix) with two completers — one app member and one free-text name — demonstrating the mixed-set case; verified by hand that every row satisfies the per-row XOR, references a `done` task with a non-null `completed_at`, and has no duplicate (task, user)/(task, name) pairs. **Fixed: Total counter flashed back in briefly when starting a first timer (2026-07-15).** The 2026-07-15 "redundant first-session Total hidden" change (below) hides the Total counter while the running entry is a task's only entry, via a `showTotal` computed gated on `runningHere && entries.length === 1`. Starting a timer could still show both counters for one render before Total disappeared. Root cause, in `useTaskTimer.fetchEntries`: the task's entries and the shared running-timer state were fetched concurrently via `Promise.all`, but the running-timer refresh wrote to its own ref the instant _its_ request resolved, independent of when the entries list finished. If that request won the race, `runningHere` flipped `true` while `entries` still held the pre-start list (length 0), so `showTotal` stayed `true` for a tick — flashing Total alongside "This session" until the entries fetch caught up and hid it. Fix: `useRunningTimer` now exposes `fetchRunningEntry()`, which fetches the running entry/title without touching state; `useTaskTimer.fetchEntries` uses it alongside `listTimeEntries` and assigns `entries` and `runningEntry` together in the same synchronous block once both requests resolve (the running-entry assignment is unguarded by the task-staleness check, same as before, since it's task-independent global state — only the `entries` assignment is guarded). `refresh()` on `useRunningTimer` is now a thin wrapper over `fetchRunningEntry()`, so its other callers (`app/layouts/default.vue`, `app/pages/index.vue`) are unaffected. No schema or UI-markup changes. `eslint`, `prettier --check`, `nuxt typecheck`, and the full 309-case Vitest suite are all clean; no new tests (a timing/ordering fix in already-tested fetch logic, not new behavior to unit test). ## Known Issues

- **Fixed:** the 2026-07-06 tag-normalization migration (`20260706150000_normalize_tag_names.sql`) failed `supabase db push` with `function min(uuid) does not exist` (SQLSTATE 42883) — Postgres has no `min`/`max` aggregate for `uuid`. The canonical-survivor CTEs now use `distinct on (farm_id, normalized_name) ... order by ... id` instead of `min(id)`. Applied to the hosted project successfully after the fix.
- **Fixed:** `/map`'s default zoom was too high (too zoomed in), then the 2026-07-06 fix over-corrected in the wrong direction — see the 2026-07-08 entry above for the final correction (also adjusted the individual-task maps' zoom levels).
- **Not a bug:** the "Use my location" report from the 2026-07-06 brainstorming pass turned out to be inaccurate GPS on the reporting device, not a `LocationPicker`/geolocation-handling issue — closed without a code change.
- **Fixed:** `useCategories.ts`/`useTasks.ts` derived the acting user's id via `user.value?.id`, but `useSupabaseUser()` returns the decoded access-token JWT payload, whose id claim is `sub` — `id` doesn't exist on that payload (TypeScript didn't catch it because `JwtPayload` has a permissive `[key: string]: any` index signature). This made `actorUserId` `undefined` for every signed-in user, every time: category creation silently no-op'd (the guard just `return`s) while task create/update/status-change/delete threw `"No active farm or signed-in user"`. This is very likely why M4/M5 hosted verification (see Next Steps) kept getting deferred — the UI would have failed immediately. Fixed by extracting a `getActorUserId` helper (`app/utils/auth.ts`) that reads `.sub`, covered by `tests/auth.test.ts`; both composables now use it.
- Vercel's PR preview build for the M3 branch initially failed with `[vite]: Rollup failed to resolve import "tslib" from ".../@supabase/functions-js/.../FunctionsClient.js"`, even though `pnpm build` passed both locally and in GitHub Actions CI on the same commit. It reproduced across multiple pushes (including one that added `tslib` as an explicit direct dependency, which didn't help) and was resolved by clearing Vercel's build cache and redeploying — pointing to a stale/corrupted cached `node_modules` on Vercel's side rather than a real dependency-resolution bug. The same class of issue showed up locally too: after a `supabase` CLI Homebrew upgrade, `pnpm dev` threw `Cannot find module 'tslib'` even though it's a locked dependency — `rm -rf node_modules && pnpm install` fixed it. Worth remembering if a build/dev-server fails on a dependency error that a fresh install elsewhere doesn't reproduce: try a clean reinstall (or cache-cleared redeploy on Vercel) before chasing a code fix.
- Mapbox access tokens don't support wildcard URL restrictions, so the token set for Preview would need to be unrestricted or per-preview-URL — for now `NUXT_PUBLIC_MAPBOX_TOKEN` is only set for Production and Development, and Preview deploys render OSM-only street tiles (the existing no-token fallback), which is harmless but worth knowing if a preview build's map looks different from prod's.

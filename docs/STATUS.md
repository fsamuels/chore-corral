**Activity-log attribution view fixed (2026-07-04).** The `farm_member_profiles` view shipped in `20260704053621` was broken in production: every select on it failed with `permission denied for table users`, so the task View page's Activity section errored for all users. ## Known Issues

- **Fixed:** the 2026-07-06 tag-normalization migration (`20260706150000_normalize_tag_names.sql`) failed `supabase db push` with `function min(uuid) does not exist` (SQLSTATE 42883) тАФ Postgres has no `min`/`max` aggregate for `uuid`. The canonical-survivor CTEs now use `distinct on (farm_id, normalized_name) ... order by ... id` instead of `min(id)`. Applied to the hosted project successfully after the fix.
- **Fixed:** `/map`'s default zoom was too high (too zoomed in) тАФ see the 2026-07-06 entry below.
- **Not a bug:** the "Use my location" report from the 2026-07-06 brainstorming pass turned out to be inaccurate GPS on the reporting device, not a `LocationPicker`/geolocation-handling issue тАФ closed without a code change.
- **Fixed:** `useCategories.ts`/`useTasks.ts` derived the acting user's id via `user.value?.id`, but `useSupabaseUser()` returns the decoded access-token JWT payload, whose id claim is `sub` тАФ `id` doesn't exist on that payload (TypeScript didn't catch it because `JwtPayload` has a permissive `[key: string]: any` index signature). This made `actorUserId` `undefined` for every signed-in user, every time: category creation silently no-op'd (the guard just `return`s) while task create/update/status-change/delete threw `"No active farm or signed-in user"`. This is very likely why M4/M5 hosted verification (see Next Steps) kept getting deferred тАФ the UI would have failed immediately. Fixed by extracting a `getActorUserId` helper (`app/utils/auth.ts`) that reads `.sub`, covered by `tests/auth.test.ts`; both composables now use it.
- Vercel's PR preview build for the M3 branch initially failed with `[vite]: Rollup failed to resolve import "tslib" from ".../@supabase/functions-js/.../FunctionsClient.js"`, even though `pnpm build` passed both locally and in GitHub Actions CI on the same commit. It reproduced across multiple pushes (including one that added `tslib` as an explicit direct dependency, which didn't help) and was resolved by clearing Vercel's build cache and redeploying тАФ pointing to a stale/corrupted cached `node_modules` on Vercel's side rather than a real dependency-resolution bug. The same class of issue showed up locally too: after a `supabase` CLI Homebrew upgrade, `pnpm dev` threw `Cannot find module 'tslib'` even though it's a locked dependency тАФ `rm -rf node_modules && pnpm install` fixed it. Worth remembering if a build/dev-server fails on a dependency error that a fresh install elsewhere doesn't reproduce: try a clean reinstall (or cache-cleared redeploy on Vercel) before chasing a code fix.
- Mapbox access tokens don't support wildcard URL restrictions, so the token set for Preview would need to be unrestricted or per-preview-URL тАФ for now `NUXT_PUBLIC_MAPBOX_TOKEN` is only set for Production and Development, and Preview deploys render OSM-only street tiles (the existing no-token fallback), which is harmless but worth knowing if a preview build's map looks different from prod's.

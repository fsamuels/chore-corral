**M8 in progress** (app-side implementation complete; storage bucket and all four per-operation RLS policies confirmed present on the hosted project via direct SQL query. End-to-end verification — actual camera capture, compression, upload, and delete — still needs a real mobile device that a sandboxed environment can't provide). Photos ride on the `task_photos` table and the private `task-photos` Storage bucket that have existed since the M2 migration — no schema change. The service layer (`app/services/photos.ts`) wraps both table and storage access: `listPhotos`, `uploadTaskPhoto` (the photo id is generated client-side since the storage path `{farm_id}/{task_id}/{photo_id}.webp` must embed it before the `task_photos` row exists; the storage upload runs before the DB insert so a failed upload never orphans a row), `deleteTaskPhoto` (same storage-then-DB ordering, in reverse), `updateTaskPhotoCaption`, and `getTaskPhotoUrl` (a short-lived signed URL, since the bucket is private) — covered by 12 new Vitest cases against a new `FakeStorageBucket` double (`tests/helpers/fake-supabase.ts`, extended with `upload`/`remove`/`createSignedUrl` and per-operation fail-injection). Client-side compression (`app/utils/photo-compression.ts`) resizes to a 1600px max edge and re-encodes to WebP via the browser Canvas API (no new npm dependency), stepping quality down from 0.82 toward a 0.5 floor until the blob is under a ~1MB target; the pure sizing math (`computeResizedDimensions`) is unit-tested, while the canvas-based encoding itself isn't (same untestable-in-happy-dom category as M7's GPS capture). No `activity_log` entries are written for photo events — photos aren't part of SPEC.md's logged-event list. The UI is a new `TaskPhotos.vue` component — camera-capture and gallery-upload buttons wired to two hidden file inputs (one with `capture="environment"`, one without), a thumbnail grid with per-photo caption editing (committed on blur) and delete, plus loading/empty/error states — backed by a `useTaskPhotos` composable that's keyed on a task id rather than the active farm (it refetches when the open task changes, not on farm switch). It's rendered only inside the `/tasks` edit dialog, not the creation dialog, since a photo's storage path requires the task to already exist.

**M7 state** (app-side implementation complete; end-to-end verification still needs a real device/GPS check that a sandboxed environment can't provide). Location rides on the `tasks.lat`/`lng` columns that have existed since the M2 migration — no schema change. The service layer carries the pin end-to-end (`TaskSummary.lat`/`lng`, create/update payloads) behind an exported `assertValidLocation` guard (both-or-neither, range-checked, `0`-safe), unit-tested with 8 new Vitest cases; location edits are unlogged field edits like notes/due date. Maps follow ARCHITECTURE.md: Leaflet + `@vue-leaflet/vue-leaflet`, with tile providers centralized in `app/utils/map-tiles.ts` — Mapbox Satellite Streets (hybrid) default with an OSM street toggle via the layer control, degrading to OSM-only when `NUXT_PUBLIC_MAPBOX_TOKEN` is unset (now set for Production/Development; Preview still falls back to OSM since Mapbox token URL restrictions don't support wildcards for per-preview-deploy URLs). Three client-only components (`.client.vue`, since Leaflet touches `window` at import): `FarmMap` (shared base map: tiles, layer control, marker-icon bundler fix, mapClick/ready events), `LocationPicker` (SPEC's capture flow — task creation auto-attempts a GPS fix shown on a mini-map for confirm/adjust by dragging the pin or tapping the map; denied/failed GPS falls back to manual placement or saving pinless; edit dialog gets the same picker without auto-capture so editing never overwrites an existing pin with the editor's position), and `TaskMap` (pins + popups). The `/map` page scopes to the active farm, fits the view to the located tasks' bounds (else the farm's `default_lat`/`default_lng`, which `useFarms` now fetches), and each pin's popup deep-links to `/tasks?task=<id>`, which opens that task's edit dialog. The Mapbox usage-tier open question in DECISIONS.md is resolved (Static Tiles free tier is ~two orders of magnitude above expected volume).

**M6 state** (app-side implementation complete; end-to-end verification still pending against the hosted project — the `/tasks` page is behind auth middleware so it can't be smoke-tested in a sandboxed environment). Tags build on the M4/M5 service-layer pattern: `app/services/tags.ts` adds `listTags` (per-farm, alphabetical), `resolveTags` (get-or-create from freeform name input — trims, drops empties, and dedupes case-insensitively in JS since Postgres `text` comparison is case-sensitive and the farm-scoped `unique(farm_id, name)` constraint needs exact-enough matching to avoid near-duplicate tags like "Fence"/"fence"), `setTaskTags` (replaces a task's full tag set: delete then bulk-insert), and `listTagsForTasks` (bulk per-task lookup via two sequential queries rather than a join, matching the fake test client's and this codebase's no-embedded-resource style). `app/services/tasks.ts` now resolves and attaches tags on `createTask`/`updateTask` and bulk-attaches them on `listTasks`, so `TaskSummary.tags` is populated end-to-end. 13 new Vitest cases in `tests/tags.test.ts` plus 4 more in `tests/tasks.test.ts` cover creation/reuse/case-insensitive dedup/farm-scoping and the create/update/list integration. The UI adds a `useTags` composable (read-only, mirrors `useCategories`) and a `v-combobox` tags field (multiple, chips, freeform autocomplete against the farm's existing tags) to both the task creation dialog's "More details" expand and the task edit dialog, plus small tag chips shown under each task's title in the `/tasks` list view.

**M5 state** (app-side implementation complete; end-to-end verification against the hosted project still pending). Task CRUD follows the M4 layering: a service layer (`app/services/tasks.ts`) wraps all Supabase access — `listTasks` (sorted urgent-first client-side with an oldest-first tiebreak via an exported `compareTasks`), `createTask`, `updateTask` (field edits only, deliberately unlogged per SPEC), `changeTaskStatus` (sets `completed_at` on done, clears it on reopen, logs the old/new pair, no-ops on a same-status call), hard `deleteTask`, and a pure `isTaskOverdue` (local-calendar-date comparison; due today ≠ overdue). Task events (`task_created` / `task_status_changed` / `task_deleted`) land in `activity_log` with the `task_title` snapshot DATA_MODEL.md requires. The lifecycle logic — an ARCHITECTURE.md priority test area — is covered by Vitest cases in `tests/tasks.test.ts` against the in-memory Supabase fake (which gained `delete()` support). Tasks referencing a soft-deleted category render "(deleted category)" rather than silently reassigning. Exercising the full lifecycle against the hosted project (now unblocked — see Next Steps) hasn't happened yet.

## Known Issues

- **Fixed:** `useCategories.ts`/`useTasks.ts` derived the acting user's id via `user.value?.id`, but `useSupabaseUser()` returns the decoded access-token JWT payload, whose id claim is `sub` — `id` doesn't exist on that payload (TypeScript didn't catch it because `JwtPayload` has a permissive `[key: string]: any` index signature). This made `actorUserId` `undefined` for every signed-in user, every time: category creation silently no-op'd (the guard just `return`s) while task create/update/status-change/delete threw `"No active farm or signed-in user"`. This is very likely why M4/M5 hosted verification (see Next Steps) kept getting deferred — the UI would have failed immediately. Fixed by extracting a `getActorUserId` helper (`app/utils/auth.ts`) that reads `.sub`, covered by `tests/auth.test.ts`; both composables now use it.
- Vercel's PR preview build for the M3 branch initially failed with `[vite]: Rollup failed to resolve import "tslib" from ".../@supabase/functions-js/.../FunctionsClient.js"`, even though `pnpm build` passed both locally and in GitHub Actions CI on the same commit. It reproduced across multiple pushes (including one that added `tslib` as an explicit direct dependency, which didn't help) and was resolved by clearing Vercel's build cache and redeploying — pointing to a stale/corrupted cached `node_modules` on Vercel's side rather than a real dependency-resolution bug. The same class of issue showed up locally too: after a `supabase` CLI Homebrew upgrade, `pnpm dev` threw `Cannot find module 'tslib'` even though it's a locked dependency — `rm -rf node_modules && pnpm install` fixed it. Worth remembering if a build/dev-server fails on a dependency error that a fresh install elsewhere doesn't reproduce: try a clean reinstall (or cache-cleared redeploy on Vercel) before chasing a code fix.
- Mapbox access tokens don't support wildcard URL restrictions, so the token set for Preview would need to be unrestricted or per-preview-URL — for now `NUXT_PUBLIC_MAPBOX_TOKEN` is only set for Production and Development, and Preview deploys render OSM-only street tiles (the existing no-token fallback), which is harmless but worth knowing if a preview build's map looks different from prod's.

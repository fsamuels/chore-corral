## Current Phase

**M8 in progress** (app-side implementation complete; end-to-end verification blocked on the same hosted-setup steps as M2‚ÄìM7, plus a real mobile device check for camera capture that a sandboxed environment can't provide). Photos ride on the `task_photos` table and the private `task-photos` Storage bucket that have existed since the M2 migration ‚Äî no schema change. The service layer (`app/services/photos.ts`) wraps both table and storage access: `listPhotos`, `uploadTaskPhoto` (the photo id is generated client-side since the storage path `{farm_id}/{task_id}/{photo_id}.webp` must embed it before the `task_photos` row exists; the storage upload runs before the DB insert so a failed upload never orphans a row), `deleteTaskPhoto` (same storage-then-DB ordering, in reverse), `updateTaskPhotoCaption`, and `getTaskPhotoUrl` (a short-lived signed URL, since the bucket is private) ‚Äî covered by 12 new Vitest cases against a new `FakeStorageBucket` double (`tests/helpers/fake-supabase.ts`, extended with `upload`/`remove`/`createSignedUrl` and per-operation fail-injection). Client-side compression (`app/utils/photo-compression.ts`) resizes to a 1600px max edge and re-encodes to WebP via the browser Canvas API (no new npm dependency), stepping quality down from 0.82 toward a 0.5 floor until the blob is under a ~1MB target; the pure sizing math (`computeResizedDimensions`) is unit-tested, while the canvas-based encoding itself isn't (same untestable-in-happy-dom category as M7's GPS capture). No `activity_log` entries are written for photo events ‚Äî photos aren't part of SPEC.md's logged-event list. The UI is a new `TaskPhotos.vue` component ‚Äî camera-capture and gallery-upload buttons wired to two hidden file inputs (one with `capture="environment"`, one without), a thumbnail grid with per-photo caption editing (committed on blur) and delete, plus loading/empty/error states ‚Äî backed by a `useTaskPhotos` composable that's keyed on a task id rather than the active farm (it refetches when the open task changes, not on farm switch). It's rendered only inside the `/tasks` edit dialog, not the creation dialog, since a photo's storage path requires the task to already exist.

**M7 state** (app-side implementation complete; end-to-end verification blocked on the same hosted-setup steps as M2‚ÄìM6, plus a real device/GPS check that a sandboxed environment can't provide). Location rides on the `tasks.lat`/`lng` columns that have existed since the M2 migration ‚Äî no schema change. The service layer carries the pin end-to-end (`TaskSummary.lat`/`lng`, create/update payloads) behind an exported `assertValidLocation` guard (both-or-neither, range-checked, `0`-safe), unit-tested with 8 new Vitest cases; location edits are unlogged field edits like notes/due date. Maps follow ARCHITECTURE.md: Leaflet + `@vue-leaflet/vue-leaflet`, with tile providers centralized in `app/utils/map-tiles.ts` ‚Äî Mapbox Satellite Streets (hybrid) default with an OSM street toggle via the layer control, degrading to OSM-only when `NUXT_PUBLIC_MAPBOX_TOKEN` is unset (token still needs to be created and wired ‚Äî see Next Steps). Three client-only components (`.client.vue`, since Leaflet touches `window` at import): `FarmMap` (shared base map: tiles, layer control, marker-icon bundler fix, mapClick/ready events), `LocationPicker` (SPEC's capture flow ‚Äî task creation auto-attempts a GPS fix shown on a mini-map for confirm/adjust by dragging the pin or tapping the map; denied/failed GPS falls back to manual placement or saving pinless; edit dialog gets the same picker without auto-capture so editing never overwrites an existing pin with the editor's position), and `TaskMap` (pins + popups). The `/map` page scopes to the active farm, fits the view to the located tasks' bounds (else the farm's `default_lat`/`default_lng`, which `useFarms` now fetches), and each pin's popup deep-links to `/tasks?task=<id>`, which opens that task's edit dialog. The Mapbox usage-tier open question in DECISIONS.md is resolved (Static Tiles free tier is ~two orders of magnitude above expected volume).

**M6 state** (app-side implementation complete; end-to-end verification blocked on the same hosted-setup steps as M2‚ÄìM5, plus a `/tasks` page that's behind auth middleware so it can't be smoke-tested in a sandboxed environment without a live Supabase project either). Tags build on the M4/M5 service-layer pattern: `app/services/tags.ts` adds `listTags` (per-farm, alphabetical), `resolveTags` (get-or-create from freeform name input ‚Äî trims, drops empties, and dedupes case-insensitively in JS since Postgres `text` comparison is case-sensitive and the farm-scoped `unique(farm_id, name)` constraint needs exact-enough matching to avoid near-duplicate tags like "Fence"/"fence"), `setTaskTags` (replaces a task's full tag set: delete then bulk-insert), and `listTagsForTasks` (bulk per-task lookup via two sequential queries rather than a join, matching the fake test client's and this codebase's no-embedded-resource style). `app/services/tasks.ts` now resolves and attaches tags on `createTask`/`updateTask` and bulk-attaches them on `listTasks`, so `TaskSummary.tags` is populated end-to-end. 13 new Vitest cases in `tests/tags.test.ts` plus 4 more in `tests/tasks.test.ts` cover creation/reuse/case-insensitive dedup/farm-scoping and the create/update/list integration. The UI adds a `useTags` composable (read-only, mirrors `useCategories`) and a `v-combobox` tags field (multiple, chips, freeform autocomplete against the farm's existing tags) to both the task creation dialog's "More details" expand and the task edit dialog, plus small tag chips shown under each task's title in the `/tasks` list view.

**M5 state** (app-side implementation complete; end-to-end verification blocked on the same hosted-setup steps as M2‚ÄìM4). Task CRUD follows the M4 layering: a service layer (`app/services/tasks.ts`) wraps all Supabase access ‚Äî `listTasks` (sorted urgent-first client-side with an oldest-first tiebreak via an exported `compareTasks`), `createTask`, `updateTask` (field edits only, deliberately unlogged per SPEC), `changeTaskStatus` (sets `completed_at` on done, clears it on reopen, logs the old/new pair, no-ops on a same-status call), hard `deleteTask`, and a pure `isTaskOverdue` (local-calendar-date comparison; due today ‚âÝ overdue). Task events (`task_created` / `task_status_changed` / `task_deleted`) land in `activity_log` with the `task_title` snapshot DATA_MODEL.md requires. The lifecycle logic ‚Äî an ARCHITECTURE.md priority test area ‚Äî is covered by Vitest cases in `tests/tasks.test.ts` against the in-memory Supabase fake (which gained `delete()` support). The UI is a `/tasks` page: a priority-ordered `v-data-table` scoped to the active farm, category filter (All / Uncategorized / per-farm categories), inline status select per row, overdue chip + red due date, strikethrough for done tasks, a creation dialog (Title/Category/Priority with a "More details" expand for Notes and Due date), an everything-expanded edit dialog hosting the delete confirm, and a `useTasks` composable that re-fetches on farm switch, plus an app-bar nav button. Tasks referencing a soft-deleted category render "(deleted category)" rather than silently reassigning. Exercising the full lifecycle against the hosted project still awaits the M2 migration + M3 OAuth setup (Next Steps 1‚Äì6).

**M4 state** (app-side implementation complete; end-to-end verification blocked on the same hosted-setup steps as M2/M3). Category management lives behind a service layer (`app/services/categories.ts`) that wraps all Supabase access: `listCategories` / `createCategory` / `deleteCategory`, with soft-deletion refused while any `not_started`/`in_progress` task references the category (surfaced as a typed blocked result, not an exception) and `category_created` / `category_deleted` events written to `activity_log` with a category-name snapshot in `event_detail`. That delete guard ‚Äî M4's called-out priority test area ‚Äî is unit-tested with 12 Vitest cases against a seeded in-memory fake of the Supabase query chains (`tests/categories.test.ts`, `tests/helpers/fake-supabase.ts`). The UI is a `/categories` page scoped to the active farm (create form, confirm-dialog deletion, blocked-delete warning, empty/loading/error states) backed by a `useCategories` composable that re-fetches on farm switch, plus an app-bar nav button. `@supabase/supabase-js` is now a direct devDependency so services can import the `SupabaseClient` type under pnpm's isolated layout. Creating/deleting real categories against the hosted project still awaits the M2 migration + M3 OAuth setup (Next Steps 1‚Äì6).

**M3 state** (M1's Vercel connection and M2's hosted migration also still pending ‚Äî see Known Issues / Next Steps): the app-side auth flow is implemented via `@nuxtjs/supabase`: a `/login` page with Google `signInWithOAuth`, the `/confirm` OAuth callback (with error/timeout states), and route protection through the module's auth middleware plus a global membership middleware. The unrecognized-login state is a dedicated `/no-access` page ‚Äî an authenticated user whose RLS-scoped `farms` query returns zero rows is redirected there and sees the invite-only explanation, their signed-in email, and a sign-out button. The farm switcher is scaffolded as an app-bar dropdown (`FarmSwitcher.vue`) backed by a `useFarms` composable with the active-farm selection persisted in a cookie; `/` is now the authenticated home showing the active farm. Not yet done (all blocked on manual/hosted setup): enabling the Google provider in the Supabase dashboard + Google Cloud OAuth client, applying the M2 migration, and seeding farms/memberships ‚Äî so no end-to-end login has been verified yet.

**M2 state:** the full MVP schema exists as a versioned Supabase CLI migration (`supabase/migrations/20260702154910_m2_schema_and_rls.sql`): both enums, all eight tables per DATA_MODEL.md, deny-by-default RLS with the farm-membership pattern on every farm-scoped table (append-only `activity_log`), and the private `task-photos` storage bucket with path-based per-operation policies. The migration and its RLS behavior were validated against a local Postgres 16 cluster with a shim of the Supabase-managed surface ‚Äî 13 impersonation tests confirm cross-farm isolation for reads, writes, and storage paths. Not yet done: the migration hasn't been applied to the hosted Supabase project, and the two farms/memberships aren't seeded, so M2's done-state isn't met yet.

Earlier M1 state: Nuxt + Vuetify app scaffolded with the full tooling set (ESLint, Prettier, vue-tsc, Husky/lint-staged, Vitest, GitHub Actions CI); `app/pages/` file-based routing with the scaffold landing card at `/` and a Vuetify component sampler at `/components-demo` (demo/reference only). ## Known Issues

- The Vercel GitHub App is now connected and PR preview deploys run, but `NUXT_PUBLIC_SUPABASE_URL`/`KEY` are still only set for the Production and Development environments, not Preview ‚Äî preview builds log `[@nuxt/supabase] WARN Missing NUXT_PUBLIC_SUPABASE_URL/KEY` as a result (harmless for now since nothing queries Supabase at build time, but needs closing out ‚Äî see Next Steps).
- Vercel's PR preview build for the M3 branch initially failed with `[vite]: Rollup failed to resolve import "tslib" from ".../@supabase/functions-js/.../FunctionsClient.js"`, even though `pnpm build` passed both locally and in GitHub Actions CI on the same commit. It reproduced across multiple pushes (including one that added `tslib` as an explicit direct dependency, which didn't help) and was resolved by clearing Vercel's build cache and redeploying ‚Äî pointing to a stale/corrupted cached `node_modules` on Vercel's side rather than a real dependency-resolution bug. Worth remembering if a future Vercel build fails on a dependency error that CI doesn't reproduce: try a cache-cleared redeploy before chasing a code fix.

# Chore Corral — Decisions

A running log of the reasoning behind non-obvious choices made during planning. This complements PR descriptions (which document the agentic development process per-change) by capturing project-level "why" in one place, rather than scattered across individual PRs.

## Open Questions (Unresolved)

Tracked here until decided. Once resolved, each moves out of this section and becomes its own dated entry below, following the same pattern used to resolve the priority-tier and `activity_log` TBDs.

### Default farm + persistence on login — resolve during M3

A user can belong to multiple farms. Nothing yet specifies which farm is active immediately after login, or whether the chosen farm persists across sessions (localStorage, a server-side preference, or always defaulting to the first membership row).

## Stack: Nuxt + Vuetify (diverging from Durak Tracker's Next.js/React)

Chosen deliberately to diversify framework depth rather than repeat the Next.js/React stack. Two factors drove this:

1. **Genuine learning value** — Vue's reactivity model and Nuxt's data-fetching conventions (loaders, composables) are architecturally distinct from React/Next, not just a syntax reskin.
2. **Portfolio credibility** — prior professional exposure to Vue (adjacent, not hands-on) meant Nuxt could convert that into demonstrable, shipped ownership, strengthening a multi-framework portfolio narrative for EM roles rather than showing depth in only one stack.

React remains the more market-dominant framework for EM hiring signal purposes, but depth across two frameworks (Next AND Nuxt) was judged a stronger signal of adaptability than repeated depth in one.

Vuetify was chosen over Nuxt UI, PrimeVue, and Quasar specifically for its Material Design foundation, given an explicit preference for Material's simplicity and mobile-friendliness, and its official, well-maintained Nuxt integration.

## Staying on Vercel + Supabase (not evaluating new hosting/DB platforms)

Explicitly ruled out setting up new accounts/infrastructure for this project. This kept the stack-selection conversation scoped to frontend framework and library choices rather than re-litigating hosting/backend platform choices already validated by Durak Tracker.

## Database: Postgres (no alternative DB considered)

Supabase's value (auto-generated APIs, RLS, Auth, Realtime, Storage) is built directly on Postgres internals — swapping the database engine isn't really separable from swapping Supabase itself, which would contradict the "no new infrastructure" constraint. Where a real decision existed, it was about Postgres _extensions_ (PostGIS for future boundary/geospatial work) rather than a different database engine entirely.

## Authorization: RLS + application-layer, deliberately combined

This was the most-discussed architectural decision in planning. The person's career background is entirely in application-layer security; Postgres RLS was genuinely unfamiliar. Two options were on the table:

- **Application-layer only** (skip RLS): more transferable skill, matches existing expertise, but loses defense-in-depth if Supabase's auto-generated REST API is ever exposed and a code path forgets a scoping check.
- **RLS as primary/only enforcement**: idiomatic Supabase pattern, real production-grade knowledge, but a net-new skill on top of an already-large pile of new things in this project (Nuxt, Vuetify, PostGIS potentially).

**Decision: use both.** RLS is enabled deny-by-default on every farm-scoped table as a backstop, while business-logic rules that don't map cleanly to declarative SQL (e.g. "can't delete a category with active tasks") live in application code. This was chosen specifically because RLS represented genuine new learning surface area the person wanted to invest in (reversing an earlier recommendation to skip it), while still leaning on application-layer judgment where that's the better tool.

## Multi-tenancy from day one (farms as top-level entity)

Even though MVP usage is just one person plus one collaborator, the data model was built multi-tenant (farms, many-to-many membership) from the very first migration, rather than single-workspace with a later retrofit. This was judged worth the small added complexity now because retrofitting multi-tenancy after the fact would touch nearly every table and RLS policy — an expensive migration to defer, given monetization was explicitly on the table as a possibility.

The person's own test/staging farm (Clarkson's Farm) doubles as a practical justification independent of monetization — multi-tenancy was needed anyway for a personal dev/test farm separate from the real one (Reign Cloud Ranch).

## Hard delete + activity log, not soft delete

Tasks are hard-deleted (a real DELETE, not a hidden flag) specifically because the person wanted true deletion available (mistakes/clutter shouldn't linger forever), while still preserving a historical trace via the activity log as a separate, independent record. This avoids the awkward middle ground of soft-deleted tasks cluttering queries indefinitely while still giving up nothing on historical visibility, since the activity log — not the task table — is the system of record for "this existed and was deleted."

## Photos: compress rather than auto-delete for storage management

Storage-cost concerns were initially raised via "delete photos on task completion" as a lever. After walking through Supabase Storage's free-tier limits (1 GB) against expected volume (<500 photos, growing), client-side compression (resize to 1600px, WebP conversion, targeting 500 KB–1 MB per photo) was judged sufficient to keep the project on the free tier indefinitely at stated scale — making deletion-on-completion unnecessary as a cost-control mechanism. Photos now persist regardless of task status; deletion strategy is deferred to ROADMAP.md only if actual usage significantly exceeds projections.

## Maps: Leaflet (not Mapbox GL JS or Google Maps SDK)

Leaflet was chosen specifically because it treats tile providers as swappable configuration (standard XYZ tile layers) rather than binding the app to one vendor's SDK. This mattered because satellite imagery provider comparison (Mapbox vs. Esri) was an explicit, stated goal — Leaflet keeps that comparison cheap. A Mapbox-GL-JS-first or Google-Maps-SDK-first choice would have made an Esri comparison a much larger rework later.

## Testing: moderate rigor (unit/component, no E2E) for MVP

Given the volume of genuinely new technology already in this project (Nuxt, Vue, Vuetify, RLS, PostGIS on the horizon), full E2E coverage was judged as spreading learning effort too thin for MVP. Unit/component testing targets the areas with real logic risk (validation schemas, status-transition logic, category-deletion guard rails) without the added Playwright learning curve. E2E is explicitly not abandoned — just sequenced after core flows stabilize, since this app has meaningfully more end-to-end-worthy flows (GPS capture, photo upload, map interaction) than Durak Tracker did.

## Documentation structure: granular from the start

Durak Tracker started as a single large spec document later split into README/ROADMAP/STATUS/ARCHITECTURE. For Chore Corral, the granular structure (SPEC, DATA_MODEL, ARCHITECTURE, ROADMAP, MILESTONES, STATUS, DECISIONS as separate files from day one) was chosen to avoid that same churn, given this project's larger scope (multi-tenancy, maps, photos, activity log) made a single document likely to become unwieldy even faster than Durak Tracker's did.

DATA_MODEL was split out from ARCHITECTURE specifically because of how much schema detail this project accumulated during planning (farms, memberships, tasks, categories, tags, activity log, photos) — enough to warrant its own home rather than crowding general architecture discussion.

MILESTONES was kept separate from ROADMAP on the reasoning that they serve different questions: ROADMAP answers "where is this headed" (open-ended, low-maintenance), while MILESTONES answers "what do I do next, in order, with a clear done-state" (scoped, sequential, actively maintained during build).

## Priority tiers: Urgent / Soon / Whenever, implemented as a Postgres enum

The tier set (SPEC.md, DATA_MODEL.md) was left as an open placeholder during initial planning. Resolved to three tiers — Urgent, Soon, Whenever — matching the casual, non-bureaucratic tone of the rest of the app (no "P0/P1/P2" enterprise-style labeling).

Implemented as a Postgres `enum` type rather than a `priorities` lookup table, since no admin tooling is planned to manage priorities as data — the spec is explicit that this list is fixed, global, and not user-editable, which is exactly the case an enum fits best. A lookup table's extra flexibility (renaming, reordering, adding tiers without a migration) isn't worth the added join for a value that's asserted to never change. If that assumption breaks later (e.g. a farm wants a 4th tier), migrating an enum to a table is a contained, well-understood change.

## `activity_log.task_id`: soft reference, not a hard FK

Tasks are hard-deleted (see the hard-delete decision above), but activity log entries must outlive the task they reference. Resolved by making `task_id` a plain nullable `uuid` column with **no FK constraint** — deletion of a task can never cascade into or orphan-error against its log entries, because the database enforces nothing there. To keep log entries meaningful without needing a join back to `tasks` (which may 404), `event_detail` always snapshots the task's title, on every event type, not just deletion.

## Storage RLS: derive `farm_id` from the object path via `storage.foldername()`

Resolved 2026-07-02 (during M2 planning). Supabase Storage policies can't join a `farm_id` column the way table policies do — `storage.objects` only exposes the path string. Since the bucket's path convention is `{farm_id}/{task_id}/{photo_id}.webp`, the policy extracts the first path segment with Supabase's `storage.foldername(name)` helper and checks it against `farm_memberships` directly:

```sql
(storage.foldername(name))[1]::uuid IN (
  SELECT farm_id FROM farm_memberships WHERE user_id = auth.uid()
)
```

No join through `tasks` is needed — the farm scoping is fully encoded in the path, which is exactly why the path convention leads with `farm_id`. This is implemented as **four separate policies** (SELECT, INSERT, UPDATE, DELETE) rather than one `FOR ALL` policy, so that `WITH CHECK` is properly enforced on writes (INSERT/UPDATE) and not just `USING` on reads — matching Supabase's documented storage-policy pattern.

The bucket is named `task-photos`, mirroring the `task_photos` metadata table. Buckets are scoped to the Supabase project (not global like S3 bucket names), and since dev/test (Clarkson's Farm) and production (Reign Cloud Ranch) share one Supabase project, both farms' photos share this single bucket, separated by the `farm_id` path prefix and these policies. The concrete SQL lives in DATA_MODEL.md's Storage section.

## Dev/test farm shares the production Supabase project

Clarkson's Farm (the personal dev/test farm) and Reign Cloud Ranch (production) live in the same Supabase project rather than separate projects, keeping infrastructure to a single set of environment variables and a single place to run migrations. This trades away hard isolation between dev iteration and production data — a buggy migration or test script run against the wrong farm's data has real blast radius, mitigated only by RLS's farm-scoping and by discipline about which farm_id you're operating against locally. Revisit this if the app moves toward real monetization or additional trusted users, where a bad dev-side mistake touching production data becomes a materially bigger deal than it is today.

## Farm-membership check: query `farms` under RLS, not `farm_memberships`

M3's unrecognized-login gate needs to answer "does this authenticated user belong to any farm?". The membership middleware answers it by selecting from `farms` and checking whether any rows come back, rather than querying `farm_memberships` directly. The RLS policy on `farms` already restricts visibility to farms the user is a member of, so the two queries are equivalent — but the `farms` result doubles as the data the farm switcher and home view need anyway (id + name), so one query serves both purposes and the result is cached in shared state for the session.

A deliberate distinction rides along with this: **zero rows means no access; a query _error_ does not.** If the farms query fails outright (schema not yet migrated, network trouble), the middleware lets the user through to the page, which surfaces the error, instead of redirecting to `/no-access`. Misreporting an infrastructure failure as "you haven't been invited" would send the user chasing the farm owner for a problem membership can't fix — relevant right now, since the M2 migration hasn't been applied to the hosted project yet.

## Overdue-flag timezone semantics: device-local calendar date, not UTC

Resolved during M5. `due_date` is a plain `date` column with no timezone, so "overdue" needed a defined reference point to avoid an off-by-one-day bug. Resolved to the device's **local** calendar date: a task is overdue only once its due date is strictly before today's local date (due-today is not overdue), computed by formatting the current moment as a local `YYYY-MM-DD` string and comparing it against `due_date` lexically. This matches how a farm worker actually experiences "today" — the alternative (comparing against UTC midnight) would flip a task overdue up to several hours early or late depending on the user's timezone relative to UTC, which reads as a bug in a single-timezone farm-ops context where there's no cross-timezone user base to serve instead.

## Task list tiebreaker sort: oldest-created-first within a priority tier

Resolved during M5. Priority is the primary sort key (Urgent → Soon → Whenever), but ties within a tier needed a defined secondary order. Resolved to `created_at` ascending — the oldest task in a tier surfaces first, on the reasoning that an aging urgent task deserves more visual priority than one just added, and this avoids the list silently reordering itself as new same-priority tasks are created (which a `created_at` descending or an unstable sort would both do). The comparator (`compareTasks` in `app/services/tasks.ts`) also breaks any remaining tie by `id` for a fully deterministic order, since a real timestamp collision — however unlikely — shouldn't leave row order undefined.

## Task deletion: confirmation dialog required, no undo

Resolved during M5. Task deletion is a real, irreversible hard `DELETE` with no soft-delete or recovery path (see the hard-delete decision above). The UI requires an explicit confirm-dialog step before deleting ("Delete “<title>”? This can't be undone."), matching the pattern already established for category soft-deletion in M4, rather than deleting immediately on the first click. Given there's no undo and the audit trail is `activity_log`-only (not a restorable history), a single accidental click permanently losing a task's title/notes/history was judged worse than the extra tap a confirmation costs.

## Mapbox usage tier: free-tier headroom is ample at expected scale

Resolved 2026-07-03 (during M7), closing the open question above the same way the Supabase Storage check was done for photos. Serving raster tiles from a Mapbox style into Leaflet (as `app/utils/map-tiles.ts` does) bills against the Static Tiles API, whose free tier is **200,000 tile requests/month** (per mapbox.com/pricing as of this writing; the more widely quoted 50,000 free "map loads" tier applies to Mapbox GL JS, not to XYZ raster consumption). Expected usage — two farms, two users, a few map/mini-map sessions a day at roughly 30–100 tiles per session — lands around a few thousand tiles a month, two orders of magnitude under the cap. Even a 10× misestimate stays comfortably free. Two mitigations ride along at zero cost: browser tile caching (repeat visits to the same farm view mostly re-serve cached tiles), and the OSM street layer fallback, which costs nothing against Mapbox at all. No paid tier or usage alerting is warranted at this scale; revisit only if the app gains real multi-tenant usage.

## Photo compression: browser Canvas API, not a compression library

Resolved during M8. Resizing to a 1600px max edge and WebP-encoding could have used an npm package (e.g. `browser-image-compression`), but the browser's native `createImageBitmap` + `canvas.toBlob('image/webp', quality)` already covers the full pipeline SPEC.md calls for, and iteratively stepping quality down to hit the ~500 KB–1 MB target is straightforward to hand-roll (`app/utils/photo-compression.ts`). Skipping the dependency avoids adding another library to audit/update for what's a few dozen lines of encoding logic, at the cost of writing — and not unit-testing, since Vitest's happy-dom environment has no real canvas/image decoding — that logic by hand rather than leaning on a maintained package's edge-case handling.

## Google OAuth configured via `supabase config push`, not the dashboard

Resolved 2026-07-03 during hosted setup. `supabase/config.toml` already had `[auth.external.google]` scaffolded with `client_id`/`secret` templated from `env(SUPABASE_AUTH_EXTERNAL_GOOGLE_CLIENT_ID)`/`env(SUPABASE_AUTH_EXTERNAL_GOOGLE_SECRET)`, so enabling the provider was `supabase config push` with those two vars set locally, rather than pasting credentials into the Supabase dashboard by hand. This also pushed `site_url` (set to the production Vercel URL) and `additional_redirect_urls` (localhost, production, and Vercel-preview `/confirm` paths) from the same file, so the whole Auth URL configuration is versioned and reproducible instead of living only as dashboard clicks nobody wrote down. The tradeoff: `config push` syncs the _entire_ `[auth]` block, so it also silently applied config.toml's other settings (e.g. MFA enroll/verify disabled, email OTP length/frequency) that happened to differ from whatever was previously set on the dashboard — worth checking `supabase config push`'s diff output before confirming if the dashboard has been hand-tweaked since the last push.

## Tag matching: case-insensitive in application code, not the database

Resolved during M6. SPEC.md's stated goal for tag autocomplete is reducing near-duplicate tags like "fence" vs. "fencing" — the same reasoning extends to pure case variants ("Fence" vs. "fence"), which autocomplete alone doesn't prevent if a user ignores the suggestion list and just types. The `tags` table's `unique(farm_id, name)` constraint is case-sensitive (`text` comparison), and Postgres `.in('name', [...])` filters are likewise case-sensitive, so neither the DB constraint nor a straightforward query can dedupe "Fence" against an existing "fence" row. `resolveTags` (`app/services/tags.ts`) instead fetches the farm's full tag list and matches candidate names against it case-insensitively in JS, preserving whichever casing already exists in the DB when reusing a tag and the first-seen casing when creating a new one. This trades a bit of query efficiency (one extra `listTags` call per resolve, already cheap at this app's per-farm tag volumes) for actually meeting the near-duplicate-reduction goal, rather than technically satisfying "autocomplete exists" while leaving a casing-based loophole.

## Theme preference: Supabase user metadata + cookie, not a database table

Resolved 2026-07-03 during the UI-improvements pass. The per-user theme choice (equipment-brand color scheme + light/dark) is stored in Supabase auth user metadata via `supabase.auth.updateUser({ data: { theme } })`, with a cookie copy (`cc-theme`) as the fast path. A dedicated `profiles`/preferences table would have needed a migration, RLS policies, and a fetch on every load for what is a single small JSON blob of pure UI state — user metadata gives per-account persistence across devices for free, and it already rides along on the session. The cookie exists so SSR can render the correct Vuetify theme immediately (no flash of the default theme) and so the choice survives on a device even if the metadata write fails; on auth-state changes the metadata value wins when the cookie is absent. If real per-user settings accumulate later (notification prefs, defaults), that's the point to graduate to a proper table — not for one theme field.

**Superseded 2026-07-09:** the multi-theme system (`useThemePreference.ts`, four equipment-brand Vuetify themes, `/settings`) was retired in favor of a single fixed "ranch" palette — see the next entry.

## Single ranch design system instead of per-user equipment-brand themes

Resolved 2026-07-09 during the ranch UI rework. The 2026-07-03 theme system gave each user a choice among four equipment-brand color schemes (Classic, John Deere, Kubota, Massey Ferguson) with light/dark variants — fun, but it fought the new visual direction: a cohesive warm cream/burnt-orange ranch look with priority-specific ring/pill colors, Zilla Slab headings, and custom card chrome that doesn't map cleanly onto four unrelated Vuetify palettes. Rather than maintain eight theme variants × every new component style, the app now ships one fixed palette defined as CSS custom properties in `app/assets/css/main.css` (with a matching single `ranch` Vuetify theme for component defaults) and no user-facing theme picker. `useThemePreference.ts`, the `cc-theme` cookie path, and `/settings` were deleted; any `theme` blobs left in user metadata are simply ignored. If real per-user settings return later (notifications, defaults), that's still the point to add a proper settings surface — but not for cosmetic palette swaps. The 2026-07-03 "metadata + cookie, not a table" call still stands for whatever settings do accumulate.

## Activity log gets an in-app UI, reversing the original MVP scoping call

Resolved 2026-07-04, during the tasks-page reorganization (dialogs → full pages). SPEC.md originally scoped the activity log as write-only from the app's perspective — viewable only by querying Supabase directly, not a user-facing feature, on the reasoning that a full audit-trail UI wasn't worth building for two trusted users. That held until the tasks workflow moved from dialogs to full pages: the new task View page needed real substance beyond what the old edit form's fields already showed, and the activity log — which already existed, already had every event it needed — was the natural fit rather than inventing new content for the page.

Showing "who" performed each action required solving a gap that hadn't come up before: `activity_log.actor_user_id` references `auth.users`, which isn't exposed via PostgREST, so the client had no way to turn an id into an email. Two options were considered: (a) a `profiles` table kept in sync with `auth.users` via a trigger, or (b) a `security_invoker` view joining `farm_memberships` to `auth.users` directly. (b) was chosen — no trigger to maintain, no sync-drift risk, and it's read-only and exactly as permissive as `farm_memberships`' own RLS already is (see DATA_MODEL.md's `farm_member_profiles` entry). A synced table would have added a second copy of `auth.users.email` to keep consistent for no real benefit at this app's scale.

**Amended 2026-07-04: the `security_invoker` variant was broken and was replaced with an owner-rights view.** In production, every select on the view failed with "permission denied for table users" (surfacing in the task View page's Activity section). The original reasoning missed two things about `security_invoker`. First, it subjects the querying user not just to the underlying tables' RLS but also to their plain table-level GRANTs — and `authenticated` has no SELECT grant on `auth.users` (deliberately so; it's why `auth.users` isn't exposed via PostgREST in the first place), so the join was rejected outright. Second, "exactly as permissive as `farm_memberships`' RLS" was the wrong target even in principle: that policy only lets a user see their _own_ membership rows, so the view could never have returned the other farm members whose actions the log needs to attribute. The replacement (migration `20260704190000`) keeps the same view shape and the no-synced-table decision, but runs with the view owner's privileges and enforces scoping in the view definition itself (`where farm_id in (select farm_id from farm_memberships where user_id = auth.uid())`, plus `security_barrier` and a SELECT grant restricted to `authenticated`) — callers see all members of farms they belong to, and nothing else.

## Tag naming convention: normalize on write, not just dedupe case-insensitively

Resolved 2026-07-06, from the M9 backlog brainstorming pass. The "Tag matching" decision above got tags to stop drifting on pure case variants, but left two related gaps: whitespace variants ("Fence Repair" vs. "fence repair") still created near-duplicates, and a newly-created tag kept whatever casing was first typed rather than a consistent form — so the farm's tag list still mixed "Fence", "fence", and "FENCE" depending on who typed first. The resolved convention is **lowercase, with spaces allowed** (e.g. "fence repair", not "Fence Repair" or "fence-repair" — hyphenation wasn't part of this pass and existing hyphenated tags in the seed data were left as-is). `resolveTags` (`app/services/tags.ts`) now runs every candidate name through a `normalizeTagName` helper (trim, collapse internal whitespace to single spaces, lowercase) before matching or inserting, so the stored `name` is always the normalized form — a behavior change from before, where a newly-created tag preserved first-seen casing.

Normalizing on write only fixes tags created from here forward; existing rows needed a backfill (migration `20260706150000_normalize_tag_names.sql`). The backfill can't just `update tags set name = normalize(name)` — two previously-distinct rows in the same farm can normalize to the same string (e.g. "Fence" and "fence " both become "fence"), which would violate `unique(farm_id, name)` partway through the update. So the migration merges colliding tags before renaming anything: group existing rows by `(farm_id, normalized_name)`; within each group, the smallest `id` is the canonical survivor; every `task_tags` link pointing at a losing tag id gets an equivalent link to the canonical id via `insert ... on conflict do nothing` (this — not a plain `update` — is what avoids a `(task_id, tag_id)` primary-key conflict when a task happens to already be tagged with both the canonical tag and a losing one, or with more than one losing variant of the same tag); the now-redundant old links and the losing tag rows are deleted; and only then are the surviving rows (merge survivors and tags that never collided) renamed to their normalized form, which is safe by that point since each is alone in its group. This can't be exercised against a real Postgres instance in this sandboxed environment (same limitation as every other migration in this project's history) — `supabase db push` plus a spot-check of the hosted `tags`/`task_tags` tables is a manual follow-up step.

## Shopping list: its own table, separate from tags/notes, separate from the future tool list, and unlogged

Resolved 2026-07-06, implementing ROADMAP.md's "Optional shopping list per task" item. Three calls rode along:

**Its own table (`task_shopping_items`), not a reuse of tags or notes.** A shopping item carries per-item state (a `checked` boolean) that neither existing mechanism can represent: tags are shared, farm-scoped labels with no per-task state beyond existence, and abusing them for "2x4 lumber" entries would pollute every task's autocomplete; the notes field is a single unstructured text blob, so check-off state would have to be encoded as text conventions ("[x] bolts") that the app can't reliably query or toggle. A dedicated child table (task-scoped like `task_photos`, cascade-deleted with the task, RLS through the parent task) matches what the feature actually is: a list of structured rows hanging off one task.

**Not unified with the future tool list.** ROADMAP.md explicitly keeps "tool list per task" a separate concept even though it's structurally near-identical, in case the two diverge later (e.g. a shopping list wanting price/store fields a tool list wouldn't need). This implementation honors that: no `kind`/`type` discriminator column was added to make one table serve both. If the tool list is built later, it gets its own table; if the two provably never diverge, merging them then is a contained migration — cheaper than un-merging a shared table after one side grows fields the other doesn't want.

**No `activity_log` events.** Adding, checking, renaming, or removing a shopping item writes nothing to `activity_log` — the same call made for photos in M8. SPEC.md's log records major events only, and shopping-list churn is descriptive task context (like captions or notes edits), not a lifecycle event. Logging every check/uncheck would bury the meaningful entries (status/priority/due-date changes) in noise.

One small mechanical note: items list in insertion order via `ORDER BY created_at ASC, id ASC`. The `id` tiebreaker — which `task_photos`' `taken_at` ordering doesn't have — is deliberate: `now()` is transaction-time, so bulk-inserted rows would share a `created_at` and have undefined relative order without it.

## Tool list: mirrors the shopping list precedent exactly, no new calls

Resolved 2026-07-08, implementing ROADMAP.md's "Optional tool list per task" item. This reaffirms rather than revisits the shopping-list decision above: `task_tools` is its own table (not a reuse of `task_shopping_items`, tags, or notes), remains unmerged with the shopping list per the explicit ROADMAP call, and writes no `activity_log` events for the same reason (descriptive task context, not a lifecycle event). No genuinely new judgment call arose in building it — the service layer, composable, and component are structural copies of the shopping-list ones with `tool`/`chainsaw`-flavored naming, down to the `ORDER BY created_at ASC, id ASC` tiebreaker and the allow-duplicates behavior on `name`.

## Photo gallery: Vuetify's built-in `v-carousel`, not a new lightbox dependency

Resolved 2026-07-06, implementing the task-page photo gallery/lightbox. No lightbox or carousel library (e.g. PhotoSwipe, Swiper, embla) was already installed, and none was added. Vuetify — already the app's only UI library — ships `v-carousel`/`v-carousel-item` with swipe support, transition animation, and slide state (`v-model`) built in, and `v-dialog fullscreen` already covers the modal-overlay chrome the way `v-dialog` does everywhere else in this app (e.g. the delete-confirm dialog on the task edit page). Combining the two gets a fullscreen, swipeable, arrow-navigable gallery with zero new dependencies to audit/update, at the cost of a plainer look than a purpose-built lightbox library (no pinch-zoom, no thumbnail strip) — acceptable for what SPEC.md asks for here (moving through a task's photos one at a time), and revisitable if richer gallery behavior is wanted later.

Left/right arrow-key navigation needed its own small workaround: `v-dialog` teleports its content to the document body, so a `@keydown` listener bound to the dialog component in the template doesn't reliably receive key events. Instead, a `window`-level `keydown` listener is added/removed via a `watch` on the dialog's open state, scoped to only exist while the gallery is actually open.

## Task estimated time: integer minutes, not an interval, and unlogged edits

Resolved 2026-07-06, implementing ROADMAP.md's "Task estimated time" item. Two calls rode along:

**`estimated_minutes integer` (nullable, `CHECK (estimated_minutes IS NULL OR estimated_minutes > 0)`), not a Postgres `interval`.** An `interval` is the semantically "correct" type for a duration, but it arrives over PostgREST as a string the app would have to parse and re-format on every read — needless friction against the hand-written types in `app/types/database.types.ts`, where an integer maps cleanly to `number | null`. The unit-in-the-name convention also sets up the future actual-time feature (ROADMAP.md's in-app timer) with an obvious sibling — `actual_minutes`, same unit, same type — so estimated vs. actual can someday sit side by side with zero conversion logic. Null is the sole "unset" sentinel: existing tasks were left null with no backfill, and the CHECK deliberately passes null so unset rows are unaffected while zero/negative values are rejected. No RLS or index change — the column rides on `tasks`' existing farm-membership `for all` policy, which is column-agnostic.

**No `activity_log` events for estimate edits.** SPEC.md's log records major events only, and its two field-edit exceptions — priority and due date — are deliberate and closed: an estimate is descriptive planning context, like notes or tags, not a lifecycle event. Changing or clearing it writes nothing to the log.

## Playwright doesn't auto-load `.env`; empty optional env vars are not the same as unset

Resolved 2026-07-09, while running the Playwright e2e harness (added 2026-07-07) for the first time. Two gotchas surfaced, both worth remembering if e2e ever mysteriously can't see env vars that are clearly present in `.env`:

**Nuxt's dev server loads `.env` automatically; Playwright's test runner does not.** `tests/e2e/global-setup.ts` reads `process.env.NUXT_PUBLIC_SUPABASE_URL` etc. directly, and since `playwright.config.ts` boots `pnpm dev` as its `webServer`, it's easy to assume the whole process tree shares one env — but `global-setup.ts` runs in Playwright's own Node process, which never touches `.env` unless told to. The fix is the standard one: `import 'dotenv/config'` at the top of `playwright.config.ts`. `dotenv` was already present as a transitive dependency (pulled in by Nuxt's own tooling), but was promoted to an explicit devDependency since the config now imports it directly rather than relying on an implicit transitive resolution.

**An env var set to an empty string is not the same as an unset one, and `??` doesn't tell them apart.** `.env` had `E2E_TEST_USER_EMAIL=` (present, empty) rather than the line simply being absent or commented out. `global-setup.ts`'s `process.env.E2E_TEST_USER_EMAIL ?? 'e2e-test@chore-corral.test'` only falls back on `null`/`undefined` — an empty string is neither, so the fallback never fired, and Supabase's Admin API rejected user creation with "Cannot create a user without either an email or phone." Fixed by removing the three empty optional-var lines from `.env` (not a code change) rather than switching the defaulting logic to `||` — an empty string is exactly this case's "not set" signal, but a future required var set to `""` should probably still fail loudly rather than silently falling back, so `??` stays the right operator for optional vars in general.

**Switched from the legacy `service_role` key to a Supabase Secret API key.** Both bypass RLS identically for the Admin API calls `global-setup.ts` needs, but the legacy `service_role` key is tied to the project's JWT secret — a leak means rotating that secret, which invalidates every user's session project-wide. The newer Secret API key (`sb_secret_...`) is independently revocable: leak one, revoke just that key, no mass logout. No code change was needed — it's passed through the same `SUPABASE_SERVICE_ROLE_KEY` env var — just a different key generated from the dashboard's newer API Keys UI.

## Task time tracking: a session log with one running timer per user, a one-way status link, and no manual entries yet

Resolved 2026-07-09, implementing ROADMAP.md's "Task actual time via in-app start/stop timer" item. The core shape was pre-decided by the roadmap entry itself (a time-entry log, `task_time_entries`, one row per start/stop session — not a single duration column on `tasks`); four calls rode along:

**Actual time is derived, never denormalized.** A task's actual time is the sum of `ended_at - started_at` across its entries, computed on read (`totalTrackedMs` in `app/services/time-entries.ts`), with a running entry counting up to now. No `actual_minutes` column was added to `tasks` — the estimated-time decision above anticipated one as a "sibling column", but once the entry log existed, a denormalized total would just be cached state to keep in sync across every start/stop/future-edit path, for a sum the app can compute from rows it already fetches.

**One running timer per user, farm-wide, with auto-stop (the Toggl model).** A running timer is a row with `ended_at IS NULL`; a partial unique index (`on (user_id) where ended_at is null`) caps it at one per user across all tasks and farms. Starting a timer on another task auto-stops the current one in `startTimer` rather than erroring — you can only physically do one thing at a time, and forcing an explicit stop first would just add friction. The index is the backstop against two clients racing the auto-stop. The alternative scopes considered (per-task-per-user uniqueness, allowing parallel timers) were rejected because they let one person's time double-count across tasks.

**One-way status link.** Starting a timer on a `not_started` task flips it to `in_progress` via the existing `changeTaskStatus` path (so the normal `task_status_changed` log entry is written); a task already `in_progress` or `done` is left alone. Stopping never touches status, and marking a task done does not auto-stop timers — the two-way variant was considered and rejected as more coupling than the feature needs. Timer start/stop itself writes no `activity_log` events (major-events-only, same call as the shopping/tool lists); the entries table is its own record.

**Manual after-the-fact entries deferred, cheaply.** "Forgot to start the timer" add/edit of entries is out of scope for this pass — but the schema already supports it (a manual entry is just a row inserted with both timestamps set), so deferring it costs no future migration, only UI work.

## In-place task editing: per-field inline edits with immediate save, incrementally replacing the Edit page

Resolved 2026-07-09, from a UX complaint: editing a task means navigating to a separate `/tasks/:id/edit` page and back, which is too much ceremony for "just change the due date". Three in-place shapes were considered:

**An edit dialog/bottom sheet** (move the existing form into a `v-dialog` over the View page) was rejected as the smallest-but-wrong fix — it kills the navigation but keeps the monolithic all-fields form, so a one-field tweak still means a full form session. **A whole-page edit-mode toggle** (the View page re-renders as an editable version of itself with Save/Cancel) was rejected because it roughly doubles the View page's script for the same monolithic-form experience, just without the route change. **Per-field inline editing won**: each field is edited where it's displayed — chips grow menus (priority, due date, category, estimated time), text swaps to an input on tap (title, notes), and the child-list components (photos, shopping list, tools) simply render editable on the View page, which their existing `readonly` prop already supports.

**Each field saves immediately on commit** (menu selection, Enter/blur), not into a pending changeset behind a Save button. This matches the one in-place precedent the View page already has — the status dropdown — and the per-field write plays cleanly with the activity log's field-level events (`task_priority_changed`/`task_due_date_changed` log exactly the change made, not a diff of a batch). The cost is no "cancel all my edits" — acceptable since every field remains individually re-editable in place.

**Saves reuse the existing full-field `updateTask`, carrying unchanged fields from the loaded task.** `updateTask` writes every field on each call; the View page builds the payload from the freshly fetched task plus the one changed field, rather than growing a partial-update variant in the service layer. This keeps the service API single-shape, at the cost of last-write-wins over the whole row if two members edit the same task concurrently — the same behavior the Edit page always had (its form also writes all fields), so nothing regresses; a field-granular partial update is a contained later change inside `updateTask` if concurrent editing ever becomes real.

**Incremental delivery, Edit page retired last.** Increments: chip menus first (highest value — the fields people tweak most — and lowest risk), then title/notes, tags, location, then rendering the photo/shopping/tool components editable on the View page, and only then deleting the Edit page (whose Delete button moves to the View page). Until that final step both paths coexist. A follow-on (ROADMAP.md) reuses the chip-menu components on `/tasks` list rows for quick-edit without opening the task.

## Default farm selection: persistent cookie plus a recent-activity fallback, not an explicit setting

Resolved 2026-07-10, from a usability complaint: a user working mostly in one farm would still land on a different, alphabetically-earlier farm after returning in a new browser session. Two things were wrong, and both were fixed rather than picking one:

**The saved-selection cookie (`cc-active-farm`) had no `maxAge`, making it a session cookie.** `useCookie`'s default in Nuxt, when no `maxAge`/`expires` is given, is a cookie with no `Expires` attribute — wiped on browser restart, not just after a period of inactivity. A manual switch via `/farm` (`setActiveFarm`) was therefore only ever "sticky" within one browser session. Fixed by giving `ACTIVE_FARM_COOKIE_MAX_AGE` an explicit one-year lifetime — a manual choice now persists until the user changes it again.

**The no-saved-selection fallback (`resolveActiveFarmId` in `app/utils/active-farm.ts`) picked `farms[0]`, which is alphabetically first** (`fetchFarms` orders by `name`) — not the farm the user actually uses. An explicit "default farm" setting was considered (a `user_preferences` table with a `default_farm_id` column) but rejected in favor of an automatic recent-activity fallback: it requires no settings UI, no new write path, and — per the roadmap of "come back and land in the right place" — automatically tracks a farm's usage pattern shifting over time in a way a one-time manual setting wouldn't, without the user having to remember to update it. The new `farm_recent_activity` view (see DATA_MODEL.md) supplies the "most recent task activity" signal, considering both a task's creation and completion time (`greatest(created_at, coalesce(completed_at, created_at))`) so a farm where existing tasks are actively being worked, not just one accumulating new tasks, still counts as active. This lookup only runs when there's no valid saved selection (`fetchFarms` checks the cookie against the fetched farm list before deciding whether to query it) — once a farm is chosen, manually or automatically, that choice is what persists in the cookie, and the recent-activity query isn't re-run on every load to second-guess it.

**Fallback order is saved cookie → most recently active farm → first farm (alphabetical) → none.** The last two tiers are unchanged from before and stay as safety nets: a farm with zero tasks has no row in `farm_recent_activity`, so a genuinely idle farm still resolves via the alphabetical tier rather than resolving to nothing.

## Home task card's circular button starts a timer, not "mark done"

Decided 2026-07-10, from a usage report that the home screen's one-tap complete-checkbox was too easy to fire by accident — a mis-tap silently marked a task done and dropped it off the list. Rather than add a confirmation step (a friction tax on every legitimate tap), the button's _job_ changed: tapping it now starts the task's timer (or stops it, if that task already holds the user's running timer). Starting is safe to fire accidentally — it's trivially reversible with a second tap, and can't lose state the way an accidental completion does.

**Marking done was not relocated to another home affordance — it moved off the home screen entirely**, to the task View page's existing segmented status control. Completion is a deliberate act worth a navigation; the home screen is for "what should I pick up next," and starting a timer _is_ picking something up. Reusing the same circular button (rather than adding a second control per card) keeps the card layout unchanged and the tap target large.

**The button doubles as the category-emoji slot.** An optional per-category emoji (see DATA_MODEL.md `categories.emoji`) renders inside the circle at rest, with a play icon fading in on hover/focus as the action affordance and a stop icon when that task's timer is running. This is why the emoji lives on the category, not the task — it's a recognizable per-category glyph shared across all that category's cards, not per-task decoration, and it gives the resting button something meaningful to show instead of an empty ring. The start/stop plumbing reuses the service layer's existing `startTimer`/`stopTimer` and the one-running-timer-per-user rule, so home-screen start behaves identically to starting from a task's View page (including auto-stopping a timer running elsewhere and flipping `not_started` → `in_progress`).

## Task completion attribution: member FK plus a free-text fallback, XOR-constrained, auto-filled on done

Resolved 2026-07-12, adding the optional "completed by" field to tasks. Three calls rode along:

**Two nullable columns, not one.** A single `completed_by uuid` FK can only credit app users, and a single `text` column loses the stable identity a membership FK gives (emails change; free text can't be resolved to a member later). The chosen shape is both: `completed_by` (uuid FK → `auth.users`, like `created_by`) for the common case, and `completed_by_name` (free text) for whoever did the work but isn't an app user — a contractor, a kid, a neighbor. A `tasks_completed_by_xor_name` CHECK keeps them mutually exclusive (both-null allowed — attribution is optional), following the `tasks_location_xor_pin` precedent exactly, including the app-layer `assertCompletedByXorName` guard so a bad combination fails with a readable message before the Postgres constraint fires. A separate "people" table (farm-scoped named non-user workers) was considered and rejected as premature — free text costs one column now and can be backfilled into such a table later if repeat non-user attribution becomes real.

**Auto-filled to the actor on done, cleared on reopen, editable in between.** Marking a task done credits whoever clicked it (`changeTaskStatus` already carries `actorUserId`), because that's overwhelmingly the right answer and typing it manually every time would make the field dead weight. It stays editable on the View page for the "I marked it done but Sam actually did it" case. Leaving done clears both columns, in step with `completed_at` (SPEC: no record of prior completion survives a reopen); `completed_by_name` is never auto-filled and is also cleared on any status change — the actor auto-fill wins over a stale free-text credit.

**Unlogged edits, and UI surfaced only on done tasks.** Attribution edits write no `activity_log` events — the log's field-edit exceptions (priority, due date) are deliberate and closed, and completion itself already logs `task_status_changed` with the actor. The "Completed by" pill only renders when the task's status is done, since the service layer guarantees the columns are null in every other status; showing an always-empty control on open tasks would just be noise.

## Weekly Progress page: completion history from `tasks.completed_at`, sliced client-side

Resolved 2026-07-12, building the `/progress` page (per-week completed-task list with stats). Two calls worth recording:

**Source of truth is `tasks.completed_at`, not the activity log.** The week's completions could be derived from `activity_log`'s `task_status_changed` → done events, which would make history durable — a task deleted or reopened later would still show in the week it was finished. Rejected for now: log-derived history needs de-duplication (a task reopened and re-done logs multiple done transitions; only the current one is "real" under SPEC's no-record-survives-a-reopen rule, so the log route would actually _contradict_ the reopen semantics, not just complicate them), and deleted tasks can't be linked or rendered beyond a title snapshot. Reading done tasks' `completed_at` keeps the page consistent with what the rest of the app says about a task and costs one simple query. The trade — history mutates when a task is deleted or reopened — matches existing behavior everywhere else (SPEC: a reopen erases the prior completion). If durable reporting ever becomes a requirement, the log data already exists to build it from.

**Fetch the farm's full done-task history; filter weeks client-side.** The natural query filters `completed_at` by range server-side (`gte`/`lt`), but the shared test fake deliberately supports only `eq`/`is`/`in` (its header says to keep it that way), and this codebase's established pattern is thin queries plus pure, unit-testable functions for the interesting logic (`compareTasks`, `partitionHomeTasks`, the task filters). So `listCompletedTasks` filters by farm and status only, and all week math — Monday-start local-timezone boundaries, half-open intervals, day grouping — lives in exported pure functions in `app/services/progress.ts` with direct Vitest coverage of the boundary cases (Sunday belonging to the prior Monday, year-boundary weeks, midnight edges). A bonus: week navigation never refetches, since every week is a slice of the already-loaded list. Fine at a family farm's scale; if a farm ever accumulates enough done tasks for the payload to hurt, adding a server-side range filter is a contained change to one function.

## Full emoji picker: bundle metadata only, a custom dialog over emoji-mart, and a Unicode-version cap

Resolved 2026-07-13, expanding the category emoji picker beyond its curated 20-suggestion grid to the full emoji catalog. Three calls worth recording:

**Emoji ship as Unicode text; only catalog _metadata_ is bundled.** An emoji is a Unicode character (or sequence) drawn by the platform's own emoji font — Apple Color Emoji, Noto, Segoe — which is why the same character looks different per device. The app bundles no artwork, and the existing `categories.emoji` column (already sized for multi-codepoint sequences) needed no change. What a full picker actually needs is the catalog: ~1,900 emoji with names, search tags, and Unicode group assignments, taken from `emojibase-data`'s English dataset. That JSON is pulled in via dynamic `import()` so it code-splits into its own async chunk, downloaded only when the full picker first opens and promise-cached module-level for the session (the cache clears on failure so a later open retries, e.g. after coming back online). The main bundle is untouched.

**A custom Vuetify-styled dialog, not emoji-mart.** emoji-mart is the battle-tested off-the-shelf picker, but it ships its own component UI and look, which would sit inside this app's `cc-*` token design system as a foreign object and need fighting via CSS overrides. The picker's actual requirements — search over label+tags, one group at a time, a recently-used row — are small enough that a bespoke dialog over the same data costs less than restyling a library, and it reuses the app's existing pill/grid visual language. The curated farm quick-row stays as the primary affordance; the catalog sits behind a "More…" tile, so the common case remains one tap.

**Unicode 17+ and component entries are filtered; no skin-tone variants offered.** The newest emoji render as tofu boxes on platform fonts that haven't caught up, so the catalog is capped at emoji version 16 (8 entries dropped today; raising the cap later is a one-constant change). Group 2 ("components" — skin-tone and hair swatches) isn't standalone-renderable and is excluded. Skin-tone variants aren't offered either — a category's single decorative glyph doesn't warrant a tone sub-picker, and the freeform field still accepts any variant pasted from an OS keyboard. Recents persist the last 24 picks in `localStorage` (`cc:emoji-recents`), SSR-guarded and best-effort (quota/security errors are swallowed; recents are a convenience, never load-bearing), and quick-row picks feed the same list.

## Progress daily activity: entries belong to their start day, and the week total means "tracked during the week"

Resolved 2026-07-14, adding per-day metrics and worked-on-task rows to the `/progress` page. Two semantic calls worth recording:

**A time entry counts entirely toward the local calendar day its timer started.** Per-day tracked time needs a rule for entries spanning local midnight (an 11pm→1am session). Clipping the entry at the day boundary is more "accurate" but adds interval math with DST edge cases for a case that barely occurs in practice — chores aren't overnight shifts, and the app's whole date layer (`isTaskOverdue`, `groupByCompletionDay`, week boundaries) already thinks in whole local calendar days rather than clock intervals. So `entryLocalDay` attributes the whole entry to its `started_at`'s local day, mirroring how a completion belongs to its `completed_at`'s local day. A useful invariant falls out: every entry belongs to exactly one day and one row, so each day's tracked total equals the sum of its rows' figures and the week total equals the sum of the day totals — no double counting, nothing dropped.

**The week "tracked" pill changed meaning: from "time on tasks completed this week" to "time tracked during the week."** The old definition pulled in work done in prior weeks (all of a task's history counted in whichever week it was finished) and ignored work on tasks not yet done — defensible when the page only listed completions, but incoherent next to per-day figures built from the entries themselves (the week pill wouldn't equal the sum of the visible day headings). The new definition sums entries whose timer started in the week, matching the day pills and crediting effort the week it happened rather than the week the task closed. Consequence to know: a task's per-row figure under its completion day shows only _that day's_ work, not its lifetime total — the lifetime total still lives on the task's own page.

## Multi-person task completion: a `task_completers` join table, row-level XOR, auto-credit only when empty

Resolved 2026-07-15, generalizing "completed by" (see the 2026-07-12 entry above) from a single credit per task to a set. Three calls worth recording:

**A join table, not array columns, and not two nullable scalars any more.** The prior shape (`tasks.completed_by` uuid XOR `tasks.completed_by_name` text) could only ever credit one person, which stopped being true once a family/farm wanted to record that two people finished a chore together. A `uuid[]` plus a `text[]` pair was considered and rejected: arrays can't carry a per-farm FK constraint on their elements, can't be indexed against duplicates without unnesting, and would need the same XOR-per-element logic hand-rolled in application code with none of Postgres's help. A join table (`task_completers`, one row per person credited) instead mirrors the `task_tags` precedent already established for task↔tag — same cascade-delete-with-parent, same no-`farm_id`-scope-through-`tasks` RLS shape, same "replace the whole set" service pattern (`setTaskCompleters`, patterned on the tags service). It also sets up future per-person Progress rollups ("who completed the most this week") as a `GROUP BY user_id` over this table, which a scalar column or array couldn't support without a rewrite anyway.

**The XOR moved from the task to the row, and mixing members with free-text names within one task is now allowed.** The old constraint said a task is credited to a member OR a free-text name, never both. That composition-level rule doesn't survive the move to a set — there's no reason a task finished by both an app member and a non-user contractor should have to pick one or drop a completer. So the CHECK became per-row (`(user_id IS NULL) <> (completer_name IS NULL)`, enforced app-side too by `assertValidCompleters`), and the task-level exclusivity was dropped entirely: a task's completer set can freely mix member rows and free-text rows. This is exactly the extension anticipated back in the original decision — free text was called a stopgap that "can be backfilled into such a table later if repeat non-user attribution becomes real," and this migration is that backfill (`20260715120000_task_completers.sql` copies both old columns into the new table before dropping them).

**Auto-credit on marking done only fires when the task has zero completers.** The old auto-fill-the-actor-on-done behavior doesn't generalize cleanly to a set: if it fired unconditionally on every done transition, re-marking an already-attributed task done (e.g. after a status round-trip, or a batch operation) would silently add the acting user as a second completer they may have had nothing to do with. Restricting the auto-credit to "only if the completer set is currently empty" preserves the original convenience for the common single-actor case while never overwriting or padding an attribution someone already entered by hand. Leaving done still clears the whole set, in step with `completed_at`, matching the original reopen semantics.

## Farm invites: email pre-authorization, not invite links or sent emails

Resolved 2026-07-18, implementing ROADMAP.md's "In-app farm invite flow" item. An invite could have been a shareable link/token (the common SaaS pattern) or a sent email with a magic link; both were considered and rejected in favor of a third, simpler shape: an owner records the invitee's Google-account email address in `farm_invites`, and whoever next signs in with that exact address is auto-joined by `accept_farm_invites()` (a security-definer function run once per session — see DATA_MODEL.md). This sidesteps needing any email-sending infrastructure (no transactional-email provider to add or configure) and any token-issuance/expiry logic (a link-based invite needs a signed token, an expiry window, and a "resend" path for a lost link) — Google's own sign-in already proves the address belongs to whoever's using it, so an exact, case-insensitive email match is sufficient authorization by itself. The tradeoff: an owner has to know the invitee's Google-account address up front (can't invite "someone, whoever clicks the link"), and the invitee gets no notification — the farm has to tell them out of band that they've been invited. Given this app's actual usage (a couple of trusted people per farm, invited by someone who already knows them), that tradeoff is a non-issue. Invites are revocable (deleting a still-pending row) and capped at one pending invite per `(farm_id, email)` via a partial unique index, so re-inviting after a revoke or after a member is removed works without manual cleanup.

## Membership writes: security-definer RPCs, a minimal owner/member role, and opening self-serve signup

Resolved 2026-07-18, implementing farm invites, membership roles, and self-serve farm creation together (migration `20260718120000_farm_invites_roles_and_signup.sql`). Three calls rode along:

**`create_farm()` and `accept_farm_invites()` as security-definer functions, not client-side INSERT policies on `farms`/`farm_memberships`.** A client-side INSERT policy letting a user insert their own membership row would need to constrain which `farm_id` and `role` that insert is allowed to carry — an easy hole to get wrong (e.g. a user inserting themselves as `'owner'` of an arbitrary farm id, or joining a farm they were never invited to). Creating a farm also needs two inserts (the farm row, then the owner's membership row) done atomically, which PostgREST's row-level API can't express as a single client transaction. Security-definer functions solve both problems: each function's body runs as one implicit transaction, and takes no caller-supplied `farm_id`/`role` for the sensitive part — a caller can only get exactly the behavior encoded in the function (join a farm you were invited to, at the role you were invited at; or create a farm and become its owner), nothing else. `farms` and `farm_memberships` keep zero client-side write policies as a result (see DATA_MODEL.md's RLS section) — the function boundary is now the only way in for either table.

**Owner/member role added now, gating invite management only — the smallest role footprint that unblocks invites.** An invite flow needs an answer to "who's allowed to invite people," and a role column was the natural mechanism. Rather than building a fuller permission model up front (who can remove members, who can delete tasks, etc. — real future ROADMAP.md items), this pass adds exactly the minimum: an `owner`/`member` enum column, checked only by `farm_invites`' RLS policies and the members page's UI gate. Every pre-existing membership row was backfilled to `'owner'`, since those members were provisioned by hand and are the de facto owners of their farms — nobody's access changes as a result of this migration. Task-level access is untouched: SPEC.md's "every member has equal, full access" still holds, deliberately, since widening that scope wasn't asked for and each further permission is its own design decision to make later.

**Opening self-serve signup (`/welcome` replaces `/no-access`).** The prior invite-only model meant an uninvited Google sign-in was a dead end — `/no-access`'s only affordance was signing back out. Once `create_farm()` existed to give invite acceptance the "join a farm atomically" building block it needed, letting any authenticated user call the same style of function to create their _own_ farm (and become its owner) was a small additional surface with an outsized usability win: no one is ever stuck waiting on someone else to remember to add them. The consequence is real and accepted: any Google account can now create an empty farm. This is fine because farm data stays fully isolated by RLS regardless (a stray signup can't see or touch anyone else's farm), and an abandoned empty farm costs nothing beyond a few unused rows — already covered by the existing "farm orphan cleanup" non-goal. `/welcome` keeps `/no-access`'s "share your signed-in email with an owner" messaging for the invited-but-not-yet-authorized case, alongside the new create-a-farm form.

## Member display names: sourced from Google profile metadata via the view, not a profiles table

Resolved 2026-07-18. Completed-by pills, activity attribution, the member picker, and the members page all showed raw email addresses, because `email` was the only human-readable field `farm_member_profiles` exposed. Three ways to get a real name were considered:

**Read it from `auth.users.raw_user_meta_data` through the view (chosen).** Supabase's GoTrue already captures the Google profile (`full_name`/`name`, avatar, etc.) into `raw_user_meta_data` on first sign-in **and refreshes it from the provider's claims on every subsequent sign-in** — so the name is already stored for every member (a membership only ever exists after at least one Google sign-in, whether provisioned by hand or via `accept_farm_invites()`), stays current if someone renames their Google account, and requires zero app-side capture code. `20260718150000_member_display_names.sql` recreates `farm_member_profiles` with a `display_name` column (`full_name` first, `name` fallback, blank-trimmed to null) — one migration, no new table, no trigger, no new write path.

**A `public.profiles` table synced by a trigger on `auth.users` (rejected for now).** The standard Supabase pattern, and the right move if/when users should be able to _edit_ their display name in-app (or set one independent of Google). Rejected here because it adds a table, a trigger on an auth-schema table, RLS policies, and a backfill — all to store a value Google already maintains for us. If user-editable names become a real ask, that table can be added later and coalesced ahead of the Google name without touching callers (they read the view/service, not the source).

**Capturing the name app-side at sign-in (rejected).** Client code writing the OAuth profile into app tables duplicates what GoTrue already does, adds a write that can be missed (stale name until the code path runs), and offers nothing over reading the metadata where it already lives.

**Label shape is an app-side concern.** The view exposes the full name; `app/utils/member-display.ts` (`memberShortLabels`) turns the farm's member list into the shortest unambiguous label per member: first name alone → first name + last initial (two Steves → "Steve A." / "Steve B.") → full name → email, falling to email/user-id outright for members with no stored name (possible for accounts that predate GoTrue's metadata capture or non-Google edge cases — email remains the universal fallback). Disambiguation is computed against the whole farm's member list everywhere (the activity service now fetches the full farm list rather than just the entries' actors) so the same person gets the same label on every surface.

**Avatars ride the same decision.** The Google profile picture lives in the same `raw_user_meta_data` blob (`avatar_url`, Supabase's normalized key, with `picture` — the raw OIDC claim — as fallback), so it's exposed through the view the same way (`20260718160000_member_avatar_urls.sql`) and rendered by a shared `MemberAvatar.vue` (generic account glyph when a member has no photo or the image fails to load — Google avatar URLs can expire). The images are hotlinked from Google's CDN, not copied: no storage, no sync job, and the URL refreshes on each sign-in along with the rest of the metadata. For the signed-in user's own chrome (the nav drawer's account header), the app reads the _same_ metadata straight from the JWT's `user_metadata` claim (`getUserDisplayName`/`getUserAvatarUrl` in `app/utils/auth.ts`) instead of querying the view — no DB round-trip, and by construction the same values the view would return.

## Chore reminders: server-scheduled Web Push, farm-wide audience, multiple reminders per chore

Resolved 2026-07-19. The ask: set a reminder on a chore for a specific date and time and get pinged on the phone ("wash the water trough while feeding tonight"). The shape of the solution was mostly forced by the platform; the genuinely open choices were settled explicitly with the user.

**Server-side scheduling is forced, not chosen.** The intuitive design — the installed PWA schedules a local notification for 7 PM — does not exist on the web: the Notification Triggers API that would have provided it was prototyped by Chrome and abandoned unshipped, and a service worker only runs when an event (like an incoming push) wakes it. So the timing authority must be a server, and the browser's only role is receiving a push at the moment of delivery.

**Scheduler: Supabase pg_cron + Edge Function, not Vercel Cron.** Reminders need minute-level precision; Vercel's Hobby tier caps cron jobs at once per day, which disqualifies it outright. pg_cron runs inside the database we already have, free-tier friendly, and its every-minute job is nearly free because `invoke_send_reminders()` bails on a cheap partial-index `EXISTS` probe unless something is actually due. The HTTP hop to the `send-reminders` Edge Function (via pg_net, authenticated by a shared secret in Vault) exists because the Web Push protocol — VAPID ES256 JWTs, RFC 8291 payload encryption — is not realistically implementable in SQL.

**Claim-then-send via `sent_at`, with stale suppression.** The function stamps `sent_at` in one conditional `UPDATE … RETURNING` before sending — Postgres row-locking makes concurrent invocations physically unable to claim the same reminder twice, trading "at-least-once" for "at-most-once" delivery on the theory that a missed nudge is cheaper than habituating users to duplicate pings. Reminders claimed more than ~an hour past due are deliberately dropped (claimed but never delivered): after downtime, a burst of long-stale "reminders" is worse than silence, since a chore reminder is only useful near its moment.

**Audience: the whole farm (user's explicit choice).** A reminder notifies every farm member with push enabled, on all their devices — chores here are shared farm work, not personal todos. The alternative (notify only the member who set the reminder) was offered and declined. Per-device opt-in is the volume control: nobody gets pings on a device where they never enabled them.

**Multiple reminders per chore (user's explicit choice), as a child table.** `task_reminders` follows the task-child-table house pattern (no `farm_id`, RLS through the parent task) rather than a single `remind_at` column on `tasks` — the user explicitly wanted "night before + hour before" style stacking, and a set-shaped feature gets a set-shaped table (the same reasoning that moved completion attribution to `task_completers`).

**`npm:web-push` in the Edge Function, not a JSR-native library.** It consumes the exact base64url keypair `npx web-push generate-vapid-keys` emits — the same public key the client hands `PushManager.subscribe()` — where the JSR alternative wants JWK-format keys, forking the key format between client and server. Its errors also carry `.statusCode`, which the 404/410 dead-subscription pruning keys off.

**The service worker stays push-only.** Offline caching remains its own roadmap item with its own unmade design decisions (cache invalidation, auth'd data, sync conflicts); bundling even a minimal cache into `sw.js` now would ship those decisions by accident. The worker exists solely to turn a push event into a notification and a tap into a navigation.

## Reminder snooze: mutate the same row, dual surfaces, and the project's first server route

Resolved 2026-07-19, same day as the reminders feature itself. The ask: snooze a fired reminder for 10 minutes or 1 hour.

**A snooze mutates the existing `task_reminders` row rather than inserting a new one.** Pushing `remind_at` out and clearing `sent_at` back to null makes the row re-fire through the exact pipeline that already exists (the every-minute due-scan re-finds it), keeps the chore's Reminders card honest (the snoozed reminder shows as upcoming again, not as a sent ghost plus a mystery duplicate), and leaves nothing to garbage-collect. The alternative — insert a fresh row per snooze, keep the original as history — was rejected as bookkeeping without a user: nobody asked "when did I originally want this?", and the activity log deliberately doesn't track reminder events anyway.

**Two fixed options (10 min / 1 hr), not a duration picker.** Matches the ask exactly, keeps the notification to its two action-button slots (Chrome's practical limit), and a snooze is by nature a coarse gesture — someone wanting a precise re-time can edit the reminder in the app.

**Two surfaces because the platforms genuinely split.** Android/desktop Chrome render notification action buttons; iOS web push ignores the `actions` option entirely — an iOS notification is title/body/tap, full stop. So the buttons live on the notification where the platform allows, and the chore's Reminders card offers the same two buttons on any already-sent reminder — which is what an iOS user lands on when they tap the notification. One row's state machine, two doors in.

**The action buttons required the project's first Nitro server route (`/api/reminders/snooze`).** A service worker handling a notification click has no Supabase client and cannot read the auth cookies from JS — but a same-origin `fetch()` from the worker carries the browser's cookies automatically. The route (`serverSupabaseClient` from `#supabase/server`) rebuilds the _user's own session_ from those cookies and performs the update **under RLS** — it adds no authorization logic of its own, no service-role key, and stays as thin as the services layer functions it mirrors. This is a deliberately narrow precedent: server routes here are for contexts that physically lack a Supabase client (a worker, maybe a future webhook), not a new default layer between the app and PostgREST. Failure UX in the worker: success replaces the notification (same tag) with a no-action confirmation so a snooze can't be snoozed; failure keeps the chore URL so the tap-through path still works.

## Status-change notes: an optional note on the status-change event, behind a confirm dialog

Resolved 2026-07-19. The ask: it's too easy to mark a chore Done by accident, and a Done move should be able to carry an optional note about how it went — with the ability to note _any_ status change eventually (In progress was added the same day; see below).

**The note lives in the `task_status_changed` event's `event_detail`, not a new column.** `activity_log.event_detail` is `jsonb not null` with no key constraints, and every status transition already writes exactly one such event — so a note is just an extra `note` key on that JSON, requiring no migration and no schema churn. The key is omitted entirely when the note is blank (trimmed to empty) rather than stored as `""` or `null`, keeping the detail object clean and letting readers `v-if`/`in`-test it directly. This also means the note is inherently tied to the transition it describes and shows up in the existing Activity timeline for free.

**The data layer is generic; the UI covers In progress and Done.** `changeTaskStatus` (and `setStatus`) accept an optional `note` for _any_ transition — the "note on every status change" goal is already wired at the service/composable/test level. The dialog was first shipped for Done alone, then extended (same day, on a follow-up ask) to the **In progress** segmented-control transition — a pure UI change, since the note plumbing already existed. A single `STATUS_NOTE_TARGETS` set (`['in_progress', 'done']`) drives which transitions open the dialog; the title/prompt/confirm-label are computed from the pending target. **Not started** stays immediate: it's a low-stakes move not worth the friction.

**Pausing the timer was deliberately left out (for now).** The follow-up ask also floated a note when the timer is paused, but pausing only closes the running `task_time_entries` row — it is _not_ a status change, so there's no `task_status_changed` event to carry a note, and timer events are deliberately never written to `activity_log` (see that decision). A pause note would therefore need a new home (a `note` column on `task_time_entries`, or a new logged event type) — not "just a UI change" — so it was explicitly deferred rather than forced into the wrong shape.

**A confirm dialog, doubling as an accidental-change guard.** Clicking **In progress** or **Done** (the latter reachable from either the segmented status control or the timer's "Complete chore" button — both funnel through the single `onStatusChange`) no longer applies immediately; it opens a dialog with an optional note field. **Cancel** leaves the chore in its existing status (the segmented control is bound to `task.status`, which is never touched until confirm, so there's nothing to revert). The guard matters most for Done, which was too easy to hit by mistake. Starting a timer still flips the task to In progress server-side without the dialog — that path is a side effect of beginning work, not a deliberate status pick, so gating it would be friction in the wrong place.
